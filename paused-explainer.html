Stage: Paused

Nebflow 自适应循环机制详解

1. 背景：为什么需要 Paused 阶段？

当 LLM Agent 面对复杂或困难的任务时，它可能陷入"循环"：反复调用相同的工具、不产生新信息、却不肯结束。 传统的解决方案是设置一个硬性的 turn 上限（例如 50 轮），但这会限制 agent 处理真正复杂任务的能力。

Nebflow 采用自适应循环（Progress-Aware Adaptive Loop）：不是按固定轮数限制，而是根据 进展检测（progress detection）动态调整约束强度。当 agent 连续多轮没有产生实质进展时， 系统逐步收紧限制，最终进入 Paused 阶段——所有工具被禁用，强制 agent 调用 finish() 给出当前最佳答案。

3. 进展检测（Progress Detection）

每轮 LLM 调用后，系统评估本轮是否"有进展"。以下任一情况都被视为有进展：

  • 调用了之前未调用过的新工具
  • 调用了相同工具但输入参数不同
  • 产生了文本回复（无工具调用，纯思考/回答）

如果一轮中所有工具调用都是完全重复的（相同工具 + 相同参数），且没有任何新信息产生， 则 stagnationCount 增加 1。

重复调用检测逻辑

系统维护一个 recentToolCalls 列表，记录最近 20 轮的工具调用（名称 + 输入哈希）。 当本轮的工具调用与列表中的任何记录完全匹配时，视为重复。 

val isDuplicate = recentToolCalls.exists { record =>
  record.name == call.name && record.inputHash == call.inputHash
}

4. Paused 阶段的行为

当 stagnationCount ≥ 9 时，agent 进入 Paused 阶段：

# 系统提示（注入到 agent 的 system prompt 中） [STAGE: PAUSED] All tools are disabled. You have been looping without making progress. Please call finish() with your current best answer. # 此时发送给 LLM 的可用工具列表： finish(answer: string) # 结束任务，给出答案 declareWait(reason: string) # 声明需要等待（不会结束）
注意：Paused 阶段并非强制终止。agent 仍然可以调用 declareWait 来声明"需要等待"。 如果 agent 认为当前确实无法继续（例如等待外部资源），系统会豁免本轮的停滞计数，不会继续加压。

5. 从 Paused 恢复

Paused 阶段不是单向的。如果用户发送新的消息，或 agent 调用 declareWait 后获得了新的上下文， 系统的停滞计数会被重置： 

// 每当检测到进展时
stagnationCount = 0
stage = Normal

// 用户发送新消息时
stagnationCount = 0
stage = Normal
progressStreak = 0

6. 完整执行流程示例

Turn 1-3: Read → Grep → Read # 有新信息，progressStreak++ Turn 4-6: Read(fileA) → Read(fileA) → Read(fileA) # 重复！stagnationCount=3 ↳ 进入 Cautious 阶段 Turn 7-9: Read → (纯文本回复) → Read # 文本回复重置 stagnation Turn 10-15: (连续6轮无进展) # stagnationCount=6 ↳ 进入 Conservative 阶段（仅 Read） Turn 16-21: (连续3轮无进展) # stagnationCount=9 ↳ 进入 Paused 阶段（所有工具禁用） Turn 22: finish("根据分析，最佳方案是...") # 任务结束

7. 代码实现

核心逻辑位于 AgentActor.scala 中的 evaluateProgress 和 stageConstraints：

private def stageConstraints(stage: AdaptiveStage): StageConstraints = stage match
  case AdaptiveStage.Normal       => StageConstraints(None, 200)
  case AdaptiveStage.Cautious     => StageConstraints(Some(keepSafeOnly), 200)
  case AdaptiveStage.Conservative => StageConstraints(Some(keepReadOnly), 150)
  case AdaptiveStage.Paused       => StageConstraints(Some(denyAllExceptFinish), 100)

private def evaluateProgress(
  freshCalls: List[ToolCall],
  duplicateResults: List[(ToolCall, ToolExecResult)],
  stageBlocked: List[(ToolCall, ToolExecResult)],
  hadTextReply: Boolean
): Boolean =
  val limitedFresh = freshCalls.filterNot(c =>
    Set("finish", "declareWait").contains(c.name)
  )
  limitedFresh.nonEmpty || hadTextReply

// 停滞计数更新
val newStagnationCount = if hadProgress then 0 else state.stagnationCount + 1
val newStage = stageFromStagnation(newStagnationCount)

8. 设计哲学

为什么不直接用硬 turn 限制？

硬限制（如"最多 50 轮"）的问题是：它惩罚了真正需要多轮思考的复杂任务。 一个正在稳步推进的 agent 不应该因为轮数到了而被强行打断。

自适应循环的哲学是：限制的不是轮数，而是"无效轮数"。 只要 agent 每轮都在产生新信息，它可以持续运行很长时间；只有当它开始原地打转时， 系统才逐步介入，最终温和地引导它结束。
